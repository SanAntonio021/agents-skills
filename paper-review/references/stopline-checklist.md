# 通用文稿润色审查与停稿清单

适用范围：论文、项目申请、专利草稿、技术报告、方案说明、汇报材料、申报书、用户文档等各类已成稿文本。

用途：作为跨对话、跨任务复用的统一审查口径，用于判断文稿是否仍需继续修改，以及新增意见属于实质问题还是风格偏好。

核心原则：停止润色的标准，不是“是否还能继续挑出问题”，而是“是否还存在会影响目标达成的实质缺陷”。

## 审查分类

文稿问题统一分为三类：

- `A 类：必须修改`
- `B 类：建议优化`
- `C 类：风格偏好`

执行原则如下：

- `A 类` 问题未清零时，不应停止润色。
- `B 类` 问题用于提升质量，但不应无限追求满分。
- `C 类` 问题不应作为继续多轮润色的主要依据。

## 新增 A 判定前提

只有同时满足以下条件时，某条问题才可算作独立的新增 `A 类`：

- [ ] 能准确定位到原文位置。
- [ ] 能映射到本清单中的某条 `A 类` 标准。
- [ ] 能说明为什么它属于 `A` 而不是 `B`。
- [ ] 不属于既有 `A 类` 的同一根因展开、换说法或补充证据。

若不满足以上条件，不应把该问题计为独立新增 `A 类`。

## 同根因去重规则

以下情况默认不算新的独立 `A 类`：

- [ ] 只是把上一轮已提出的 `A 类` 拆得更细。
- [ ] 只是对同一问题换了一种表述方式。
- [ ] 只是将同一根因在另一段文字上再次定位。
- [ ] 只是为同一根因补充更多例子、证据或影响说明。

上述情况应合并到原有 `A 类` 条目，或标记为“同根因补充”。只有在存在明显最高优先级的单个阻塞项时，才在 `A 类` 首条额外标记 `优先处理`。

## 跨对话注意事项

跨新对话复用本清单时，只有在当前上下文同时给出了上一轮 `A 类` 清单或可比对摘要，才适合判断“是否新增 `A 类`”。若当前任务只是普通审查当前版本，而不是比较上一轮，则不必主动讨论前序 `A 类` 清单是否缺失。

如果未提供上一轮 `A 类` 清单，则：

- [ ] 不应声称“新增 A 类问题”。
- [ ] 也不应声称“未新增 A 类问题”。
- [ ] 只能判断当前版本是否仍存在 `A 类` 问题。
- [ ] 如需严格比较新增与否，应先提供上一轮 `A 类` 列表或其摘要。
- [ ] 若用户未要求比较上一轮，不应反复输出“缺少前序 A 类清单”的提醒。

## A 类：必须修改

以下问题属于实质缺陷，应继续修改：

- [ ] 文本目标不清，受众、用途或交付目标与正文写法明显不匹配。
- [ ] 核心观点、结论、主张或方案前后矛盾。
- [ ] 关键事实、数据、术语、定义、编号、图表引用或名称不一致。
- [ ] 重要结论缺少必要依据、证据、推理链或上下文支撑。
- [ ] 明显存在事实错误、逻辑错误、因果倒置或过度推断。
- [ ] 关键段落缺失，导致读者无法理解主线、方法、结论或执行路径。
- [ ] 关键表述存在歧义，且可能引发误解、误判或错误执行。
- [ ] 结构性问题已经影响阅读，例如顺序错乱到读者无法正常跟进。
- [ ] 存在占位符、未完成句、截断段、待补材料或明显漏项。
- [ ] 存在会带来明显风险的问题，例如合规风险、承诺过度、证据越界或自我否定式表述。

## B 类：建议优化

以下问题通常值得处理，但不应成为无限打磨的理由：

- [ ] 段落顺序基本合理，但局部衔接仍可更顺。
- [ ] 句子偏长、信息密度过高，导致理解成本偏高。
- [ ] 存在重复表达、模板腔、空泛套话或导航腔。
- [ ] 同一概念有多种近义说法，尚可进一步统一。
- [ ] 个别段落重点不够突出，主次关系可以更清楚。
- [ ] 个别例子、限定语或解释略显冗长，可以压缩。
- [ ] 行文节奏不够稳定，有的段落过紧，有的段落过松。
- [ ] 局部标题、小结、过渡句还能更贴合全文结构。

## C 类：风格偏好

以下问题通常不应成为继续润色的充分理由：

- [ ] 只是另一种说法“也许更自然”。
- [ ] 只是句子听起来还能更漂亮、更像某种文风。
- [ ] 只是不同审阅者对语气、用词、节奏的个人偏好不同。
- [ ] 修改不会提升准确性、可读性或交付效果，只是表达口味变化。
- [ ] 不同版本都成立，但无法证明新版本明显更好。

## 停稿判定规则

满足以下条件时，通常应停止润色：

- [ ] `A 类` 问题已经清零。
- [ ] `B 类` 问题只剩零散项，且不影响正常理解和正式交付。
- [ ] 连续两轮审阅没有新增 `A 类` 问题。
- [ ] 新增意见主要集中在 `C 类`，而不是实质内容。
- [ ] 不同审阅意见开始互相冲突，说明已进入风格拉扯阶段。
- [ ] 继续修改的收益，小于重新引入错误、打乱稳定表述或拖延交付的风险。

默认停稿条件：

- 前 3 条全部满足；
- 后 3 条至少满足 2 条。

补充说明：

- 若当前无法判断“是否新增 `A 类`”，应改用“当前版本是否仍存在 `A 类`”来决定是否继续修改。
- 是否继续修改，优先看“未闭环的独立根因数”，而不是看表面上被列出了多少条 `A 类` 句子。

## 推荐轮次

通用情况下，建议把润色控制在 3 轮内：

1. 第一轮：只抓 `A 类` 问题。
2. 第二轮：集中处理术语、结构、重复和局部可读性问题。
3. 第三轮：只做必要压缩和统一，不再追求“更好听”。

如果第三轮之后仍在新增 `A 类` 问题，应优先判断上游内容是否尚未定稿，而不是继续把问题视为单纯润色问题。

## 快速判定问题

在判断某条修改意见是否值得继续处理时，可使用以下问题进行筛选：

- 该修改是在消除实质风险，还是仅仅替换表达方式？
- 不修改该处，是否会影响读者正确理解、信任或执行？
- 该意见属于客观缺陷，还是审阅者个人文风偏好？
- 该轮修改是在提高质量，还是在追求不存在的“无可挑剔”？

如果结论连续偏向后者，应视为进入停稿区间。

## 标准审阅提示词

以下提示词可在任意新对话中直接引用：

```text
请以“通用文稿润色审查与停稿清单”为唯一审查口径，对下述文稿进行审查。

输出时请严格分为三类：
A 类：会影响目标达成、准确性、逻辑一致性、证据支撑、结构完整性的问题；
B 类：可提升可读性和表达质量，但不改也不影响正式交付的问题；
C 类：纯风格偏好或可改可不改的问题。

请优先指出 A 类问题，不要把纯措辞偏好包装成严重问题。
如果当前上下文没有提供上一轮 A 类清单，请不要使用“新增 A 类问题”这一表述，只判断当前版本是否仍有 A 类问题。
如果当前版本没有 A 类问题，请明确写出“建议停止润色”。
```

## 说明

本清单的目标，是将“是否继续修改”转化为可重复执行的判断标准，而不是追求“无法再被提出任何意见”的状态。后者通常并不存在，也不适合作为停稿依据。
