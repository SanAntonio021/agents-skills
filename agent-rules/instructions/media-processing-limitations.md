# Media Processing Limitations

## Purpose

说明当前环境处理音频、视频和其他大体积媒体文件时的真实边界，避免把不存在的多模态能力写成默认能力。

## Scope

这里讨论的是：

- 音频转写
- 视频内容提取
- 大体积媒体文件的本地处理边界

这里不讨论普通文本、图片或已经转成文本的会议材料。

## Current Limitation

当前维护基线里，不应假设智能体天然具备以下能力：

- 直接读取大型音频或视频二进制并理解内容
- 直接播放媒体文件并做原生听写
- 在没有外部工具或外部服务时完成可靠转写

因此，面对 `.aac`、`.mp3`、`.wav`、`.mp4`、`.m4a`、`.mov` 等文件时，不要把“可处理”当成默认前提。

## Default Handling Path

1. 先确认用户要的是：
   - 原始媒体转写
   - 已有转写文本的整理
   - 视频内容摘要
2. 再确认当前环境是否已有可用工具或服务：
   - 本地 CLI，例如 `ffmpeg`、`whisper` 或其他已装工具
   - 可调用的外部 API 或现成平台
3. 如果没有现成处理链，直接说明物理边界，不伪装成即将完成。
4. 给出可推进的替代路径，而不是停在“做不了”。

## Recommended Fallbacks

### A. 用户先提供转写文本

如果用户能先用桌面端或外部服务把媒体转成文本、字幕或会议记录，再交给对应文本 skill 继续处理，通常是最稳的路径。

例如后续可转入：

- `meeting-notes`
- 其他文本摘要或整理流程

### B. 先确认本地工具链

如果用户明确要在本地完成处理，先检查是否已有：

- `ffmpeg`
- `whisper` 或同类转写 CLI
- 其他用户明确指定的媒体处理工具

只有工具真实存在时，才继续设计本地脚本或命令流程。

### C. 外部服务转写后再接手

如果用户已有可用 API 或外部平台，先把媒体转成文本，再回到当前文本工作流。

## Boundary

- 不把不存在的媒体理解能力写进普通 skill 的默认描述。
- 不用文本读取方式硬读二进制媒体文件。
- 不在没有工具链时承诺可直接完成大型媒体转写。
- 不把“图片可看”误推成“音视频也可直接处理”。

## Maintenance

- 当本机长期稳定安装了新的媒体处理工具链时，再更新本文件。
- 如果未来出现稳定可复用的媒体工作流，应优先拆成独立 skill，而不是继续堆在 handbook 说明里。
