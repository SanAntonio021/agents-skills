# 工作流契约

## Shell 传输格式

下面的提示词模板为了便于人读而使用多行格式。交给 `codex:codex-rescue` 后，subagent
必须在唯一的 companion Bash 调用前将模板序列化成一个单行参数：

- 使用“字段名=值；”分隔各字段；
- 完整保留目标、范围、证据、计划、交付物、验收标准、安全边界和输出契约；
- Bash `command` 字符串中不得出现字面 CR/LF；
- Windows 的任务参数内部不使用 XML 结束标签或 `C:/` 绝对路径；优先写相对当前
  Codex cwd 的路径，必须写绝对路径时使用反斜杠；
- 不用临时文件、here-doc、管道、重定向或命令替换；
- 计划复核、执行和返工三次调用都遵守同一规则。

单行示例：

```text
模式=只读复核，不修改文件，不执行计划；目标=...；范围=...；证据=...；计划=...；交付物=...；验收标准=...；安全边界=...；输出契约=严格按 PLAN_REVIEW 输出
```

## 计划复核提示词

首次调用 `codex:codex-rescue` 时使用以下结构，并带 `--fresh --wait`。明确写出
“只读复核，不修改文件，不执行计划”。

```text
只读复核，不修改文件，不执行计划。

任务目标：<目标>
范围与排除项：<范围>
输入与证据：<材料>
Claude 的计划：<步骤>
交付物：<交付物>
验收标准：<逐条标准>
安全边界：<限制>

请检查完整性、可执行性、证据边界、遗漏风险和验收标准。严格按 PLAN_REVIEW 输出。
```

## PLAN_REVIEW

```text
PLAN_REVIEW
结论：通过 | 需修改 | 实质分歧
已确认事项：
- ...
问题与理由：
- <问题；理由；证据或待核事实>
必须修改：
- ...
剩余风险：
- ...
```

`通过` 仍需写剩余风险；没有问题时写“无”。`需修改` 必须给可执行修改项。
涉及方向、价值判断或证据无法消除的冲突时用 `实质分歧`。

## 执行提示词

计划通过后使用同一 thread，带 `--resume --wait --write`：

```text
按已通过的最终计划执行任务。你可以在授权范围内读取和修改文件、运行命令、
生成交付物。不要扩大范围。遇到删除、重置、权限变更、付费、对外发送、硬件
操作、覆盖、会丢失既有内容的整文件替换，或计划外方向选择时停下说明，
不要自行决定。计划已经授权的局部修改不需要重复询问。

最终计划：<计划>
允许修改范围：<路径或对象>
验收标准：<逐条标准>

完成时报告实际修改文件、实际运行命令、逐条验收证据、失败项和剩余风险。
```

## 返工提示词

返工继续同一 thread，带 `--resume --wait --write`：

```text
Claude 验收未通过。请只修复以下问题，不扩大范围；修复后重新运行必要验证。

问题：
- <文件或对象；观察到的事实；为什么不通过；通过判据>

请报告修改、验证证据和仍未解决的事项。
```

## DISAGREEMENT_REPORT

```text
# 需要用户裁决

## 已达成共识
- ...

## Claude 的判断
- 结论：...
- 理由：...
- 证据：...

## Codex 的判断
- 结论：...
- 理由：...
- 证据：...

## 实质分歧
- ...

## 需要决定
- <一个明确问题及各选项影响>
```

## CODEX_FAILURE_REPORT

```text
# Codex 协作已暂停

- 失败阶段：计划复核 | 执行 | 返工
- 关键错误：<最短决定性错误>
- job ID：<已知 ID；未知则写未知>
- expectedThreadId：<已记录 ID；未记录则写未知>
- candidateThreadId：<当前候选 ID；查不到则写未知>
- 已完成内容：<文件、结果或无>
- 未完成内容：<范围>
- 恢复条件：<登录、额度、重试或可靠续接条件>
```

输出后停止，不由 Claude 继续执行。
