# 门户接口发现与安全读取

## 1. 工具顺序

任何联网操作先加载 `web-access` skill。优先顺序：

1. 站点公开 API 或文档
2. 已登录浏览器的后台网络请求
3. 浏览器页面读取
4. 必须依赖可视界面时才使用桌面控制

优先复用现有登录会话。不要索要或导出密码、Cookie、JWT 和 API key。

## 2. 先看页面触发的真实请求

在控制台正常打开价格、日志、钱包和充值页面，观察对应 XHR/fetch：

- 请求方法和路径形状
- 分页参数、时间参数和时区
- 响应 JSON 的数据列表、总数和单位字段
- 页面是否在前端二次计算倍率或货币

只记录脱敏后的路径和字段名。不要记录 request headers、完整 query string 或带令牌的 iframe URL。

## 3. 常见接口类别

下面只用于发现，不代表每个站都存在：

- 价格/模型：`pricing`、`models`、`ratio`、`group`
- 当前账户：`user/self`、`profile`、`subscription`
- 请求日志：`log/self`、`usage`、`billing/records`
- 钱包账本：`wallet`、`balance`、`ledger`
- 充值记录：`topup`、`payment`、`orders`

One API/New API 衍生站点经常保留相似路由，但字段、权限和计费逻辑可能已被修改。以当前站点响应为准，不凭产品外观套接口。

## 4. 常见日志字段

优先搜索这些字段及其同义字段：

```text
request_id, created_at, model_name,
prompt_tokens, input_tokens, cached_tokens, cache_read_tokens,
cache_creation_tokens, completion_tokens, output_tokens,
quota, cost, amount, multiplier, model_ratio, group_ratio,
cache_ratio, completion_ratio, channel, token_name, content, detail
```

`content`、`detail`、`metadata` 常是嵌套 JSON 或 JSON 字符串。使用结构化解析，不用字符串切割。先判断 `cost`/`amount` 是基础成本、最终扣费还是展示值。

## 5. 分页与时间边界

- 使用接口返回的总数或 next cursor，直到覆盖完整时间段。
- 保存原始时间戳和站点时区，再换算用户时区。
- 检查起止时间是闭区间还是半开区间。
- 用 request ID 去重；没有 request ID 时用时间、模型、Token 和扣费组合标记疑似重复。
- 页面只展示最近记录时，不把“未显示”当成“没有消费”。

## 6. 历史倍率与动态回显

常见风险：页面用当前账户倍率重新渲染旧日志，但账本扣费在请求发生时已经冻结。

检查顺序：

1. 找请求响应里的 `group_ratio`、`model_ratio`、`multiplier` 或公式说明。
2. 对比不同日期的原始响应，而不是只对比页面文本。
3. 用请求级最终扣费反推有效倍率，并核对站内单位。
4. 若只有当前倍率，没有历史倍率或请求级扣费，历史倍率写 `not_evaluable`。

不要因为多个旧记录都显示相同倍率，就自动认定历史倍率从未变化。

## 7. 账本与充值

分别获取：

- 目标时段期初和期末余额
- 请求消费流水
- 充值支付金额和实际到账 credit
- 赠送、退款、人工调整、过期或转账

订单状态必须是成功或已到账。不要把待支付、失败、已取消订单计入充值成本。输出时隐藏完整订单号和支付标识。

## 8. 只读安全规则

- 默认只调用 GET 或站点明确标注为查询的 POST。
- 不重放充值、退款、密钥创建、配置保存等写请求。
- 不把认证信息复制到 shell 历史、脚本参数、临时文件或报告。
- 响应中若含密钥、邮箱、手机号、IP 或完整 request content，只保留核账需要的脱敏字段。
- 保存报告时，先生成字段白名单；不要先保存完整响应再做脱敏。

## 9. 最小付费探针

历史证据不足时，先向用户说明预计请求数、模型和最高费用。得到明确批准后：

- 使用用户指定的测试密钥或已授权会话
- 固定短提示和最大输出 Token
- 每站测试前后记录余额
- 保存 request ID 和 usage，不保存认证信息
- 检查新日志出现后立即停止

如果余额精度不足以观测单次请求，不能擅自增加请求数量；先报告测量分辨率问题。

## 10. 失败处理

- `401/403`：先确认登录会话和页面权限，不尝试绕过权限。
- 空列表：检查分页、时间戳、时区、日志类型和保留期；一次空响应不能证明无消费。
- 字段含义不清：查看页面计算逻辑、公开文档或同一请求的账本记录。
- 站点不可达：保留已取得证据，明确当前无法实时验证。
- 反爬或验证码：切到用户正常浏览器会话；不要规避安全机制。
