# 本地技能查找与复核

## 这份技能做什么

外部市场里的搜索、比较和安装，仍然优先交给官方 `find-skills`。

这份技能负责把结果放回当前机器的实际环境里再判断一遍：

- 本地有没有已经能用的同类技能
- 该给用户上游仓库，还是给安装命令
- 这次更适合用现成技能，还是更适合改本地自建技能

不要把市场里搜到的结果一股脑都当成“直接推荐”。

## 本地规则

1. 开始查找前，先看当前会话默认可用的技能，确认本地是不是已经有同名或同类技能，避免重复安装。
2. 如果要核对本地到底装没装过，再看 `C:\Users\SanAn\.cc-switch\skills\`。
3. 如果后面要正式修改某个自建技能，再去 `D:\BaiduSyncdisk\.agents\agents-skills-src\` 找源码目录；不要直接改同步产物目录。
4. 如果需求本来就是“做一个自建技能”，就直接走当前 cc-switch 管理的自建技能仓，不要装到全局市场目录。
5. 如果拿不准该不该新装，先查现有技能，再决定要不要推荐外部技能。

## 查找前先收窄需求

如果用户要找的是内容处理、写作、审稿、改写、搜索或自动化类技能，先问最少但关键的问题，不要一上来就按关键词甩一串市场结果。

优先确认这几件事，因为它们会直接改变推荐结果：

1. 用户要的是“只指出问题”，还是“指出问题后自动改写”。
2. 输入对象是什么，例如论文、申报书、技术文档、公众号文章、营销文案、代码文档或聊天回复。
3. 用户要的是现成上游、安装命令、本地自建技能，还是只要 GitHub 仓库，方便后续用 cc-switch 管理。

如果用户已经明确说“直接给上游”“我会用 cc-switch 管理”“不要安装”，就不要继续追问安装位置，直接给仓库地址和来源页面。

## 只要现成上游时

当用户只要现成技能或上游仓库时，输出重点放在“来源清不清楚、好不好管理”，不是替用户安装：

1. 先说明本地有没有高重合技能，为什么不直接复用，或者为什么仍然建议看外部上游。
2. 再列外部候选，优先给 GitHub 仓库，其次给 skills.sh 页面。
3. 每个候选都说明“适合什么”和“要注意什么”，不要只按安装量排序。
4. 如果候选属于 humanizer、proofreader 或 style 这类写作技能，要提醒它可能会改变语气、立场或正式程度。
5. 如果用户明确用 cc-switch 管理技能，默认不要执行 `npx skills add`，只给上游仓库和必要的调用约束。

## 正式文稿类候选判断

论文、申报书和技术文档，不等同于普通的“去 AI 味”场景。推荐 humanizer 或改写类技能时，要特别检查它会不会带来这些副作用：

- 过度口语化、加入第一人称或情绪化表达。
- 为了“更自然”而弱化术语、证据边界或定量表述。
- 把正式技术判断改成营销腔、故事腔或泛泛价值表达。
- 只清理表层套话，却没有处理逻辑模糊、概念未定义、结论跳跃和中英混杂。

给这类用户推荐候选时，建议顺手附上一句可直接复制的调用约束：

```text
保持正式、客观、技术文档口吻；指出问题后再改写；不要加入第一人称、口语化表达、额外情绪或未给出的事实。
```

## 输出模板

除非用户明确要求更短版本，否则始终按下面模板输出：

```text
结论：
- <优先复用本地现成技能 / 推荐外部上游 / 更适合修改本地自建技能 / 暂不建议安装>

本地情况：
- <已有 / 没有 / 有但不适合>

首选上游：
- <GitHub 仓库>
- <skills.sh 页面>
- 适合：...
- 注意：...

备选：
- ...

建议调用约束：
<如果需要，给一句可复制的约束>
```

如果用户明确只要仓库地址，就压缩成“仓库地址 + 一句理由”。
如果本地已经有高重合技能，`结论` 里要直接写清楚是“先复用”还是“仍建议看外部上游”，不要只把信息铺在后文里让用户自己猜。

## 真实案例

如果用户要找一个技能，让 AI 检查并改写文字里的中英混杂、逻辑模糊、自造词、黑话，以及其他常见的 AI 写作问题；后续又确认文本类型是论文、申报书和技术文档，并且用户只要现成上游、会用 cc-switch 管理：

这类场景应优先推荐中文 humanizer 或 AI 写作痕迹处理类上游，例如：

- `https://github.com/op7418/Humanizer-zh`
- `https://skills.sh/op7418/humanizer-zh/humanizer-zh`

同时提醒：正式文稿使用时要加上“保持正式、客观、技术文档口吻”这类约束，避免把论文或申报书改成口语化文本。
