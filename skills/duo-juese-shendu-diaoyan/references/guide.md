# 多角色调研编排与模拟多 Agent 执行指南

本指南定义 `duo-juese-shendu-diaoyan` 在当前 Codex 环境中的默认执行方式。目标不是虚构一个并不存在的原生子代理系统，而是用“分阶段拆解 + 独立 worker 提示词 + 显式 handoff + 有约束的并行化”来模拟多 agent 协作。

## 1. 模式选择

先做能力检查：

- 如果当前会话明确提供原生 `Task` / sub-agent 工具，可以进入原生模式。
- 如果没有看到该类工具，默认使用模拟模式。
- 在模拟模式下，不要说“我已经开了三个子代理”；应说“我按三个 worker 阶段并行或串行执行”。

默认假设：

- 当前 Codex 更常见的是“可并行工具调用 + 单会话协调者”，不是“多独立上下文子会话”。
- 因此本 skill 的主路径应始终能在单会话内完成。

## 2. 何时值得启用本 Skill

只有在以下条件至少满足两个时，才值得启用：

- 研究问题天然分成两条及以上相互独立的取证支路。
- 目标证据存在明显层级，需要做降档搜索。
- 需要把碎片证据桥接成因果链、论证链或指标论证。
- 最终还要转换成申报、论文、综述、技术说明等正式文本。

如果任务只是“查一个事实”“补一个引用”“写一段普通说明”，优先用单智能体技能。

## 3. 推荐阶段拆分

最常用的四阶段：

| 阶段 | 目标 | 是否可并行 |
| --- | --- | --- |
| 窄域检索 | 在目标场景内穷搜直接证据 | 可与宽域基线并行 |
| 宽域基线 | 脱离特定场景，找通用标尺或行业基线 | 可与窄域检索并行 |
| 逻辑桥接 | 把碎片证据焊接成论证链并说明边界 | 串行 |
| 写作/审阅 | 把 bridge 包转换成正式输出 | 串行 |

可选扩展阶段：

- 反证阶段：专门找会削弱主张的证据。
- 术语归一阶段：统一指标口径、场景定义、比较基准。
- 审阅阶段：检查是否把桥接推断写成了直接事实。

## 4. 并行与串行判据

允许并行的情况：

- 两条支路的问题不同，且不依赖彼此中间结论。
- 一个支路搜直接证据，另一个支路搜通用基线。
- 多个文件读取、页面抓取、数据库查询彼此独立。

必须串行的情况：

- 下游需要上游结论来决定搜索词或判断标准。
- 需要解决证据冲突、统一口径或做因果桥接。
- 已经进入写作、判断、审稿回复、指标定性定量转换。

一句话规则：

能并行的是“取证”，应串行的是“解释、裁决、执笔”。

## 5. Worker 提示词模板

模拟模式下，每个 worker 都要有独立角色卡，避免上下文污染。

### 5.1 检索 Worker

适用：窄域检索、宽域基线、反证搜索。

模板：

```text
你现在是一个单职责 research worker。

你的唯一任务：
<一句话问题>

搜索边界：
- 必须覆盖：<范围>
- 禁止扩展到：<排除范围>
- 如果直接证据不足，允许降档到：<降档层级>

输出要求：
1. 问题重述
2. 搜索路径
3. 直接证据清单（每条含来源）
4. 间接证据清单（每条含来源，并标明为何只能算间接）
5. 冲突或缺口
6. 交给下一阶段的结论与保留项

不要写成正式成稿，不要替下一阶段做论证润色。
```

### 5.2 Bridge Worker

适用：桥接论证、指标合理性说明、首创性论证。

模板：

```text
你现在是 bridge worker，只负责把上游证据接成论证链。

输入包：
- 窄域检索结果：<粘贴或引用>
- 宽域基线结果：<粘贴或引用>

你的任务：
1. 区分直接证据、间接证据、推断。
2. 说明这些证据能支持到什么强度，不能支持到什么强度。
3. 当出现空白时，明确空白本身是否构成可用结论。
4. 输出给写作阶段的 bridge 包。

输出要求：
1. 已确认主张
2. 可桥接主张
3. 不能主张的内容
4. 证据冲突与处理
5. 仍需补证项
6. 写作阶段可直接复用的表述骨架
```

### 5.3 Writing Worker

适用：综述、申报段落、技术说明、审稿回复草稿。

模板：

```text
你现在是 writing worker。

你只能使用 bridge 包中的结论与边界，不得自行补造证据。

输入：
- bridge 包：<粘贴或引用>
- 目标文体：<申报/综述/SCI/答辩材料/其他>
- 目标受众：<谁看>

输出要求：
1. 只把已确认主张写成肯定句。
2. 把桥接推断与直接证据明确区分。
3. 对空白和保留项保留边界表达。
4. 如需引用，保持来源锚点可回溯。
```

## 6. Handoff 标准格式

每个阶段结束后，优先输出同一套 handoff 结构：

```text
[Stage]
<阶段名>

[Question]
<本阶段唯一问题>

[Scope]
<覆盖范围>

[Method]
<搜索词、筛选逻辑、降档规则>

[Direct Evidence]
- <要点 + 来源>

[Indirect Evidence]
- <要点 + 来源 + 间接原因>

[Conflicts]
- <冲突点>

[Gaps]
- <缺口或空白层级>

[Carry Forward]
- <下一阶段可直接消费的结论>
- <下一阶段必须保留的限制>
```

不要省略 `Gaps` 和 `Carry Forward`。它们是模拟多 agent 可控性的关键。

## 7. 阶梯降档检索法

当目标场景极其细分时，采用严格的降档顺序：

1. 先找目标场景内的真实系统级或实测证据。
2. 找不到则降档到相近系统的实验或工程验证。
3. 再找不到则降档到仿真、理论建模或部件级证据。
4. 最后才允许脱离目标场景，检索通用工业基线。

约束：

- 只有上一级明确不足，才进入下一级。
- 每一级都必须显式写出“找到了什么”或“没有找到什么”。
- “没有找到”本身可以成为后续首创性、空白性或风险说明的一部分，但不能夸大成已被证明。

## 8. 与其他 Skills 的配合

优先复用通用技能，不为单项目临时造 skill：

- 需要深搜底座时，优先调用 `tooluniverse-literature-deep-research` 或 `deep-research`。
- 需要桥接、批判、反证时，优先调用 `zixun-pipan-zhibi`。
- 需要正式学术写作时，转交 `proposal-sci-collab` 或更具体的写作 skill。

项目特化信息应作为输入包、临时文档或首帧上下文注入，而不是固化进本 skill。

## 9. Web AI Deep Research 提示词检查清单

为网页版 Deep Research 编写提示词时：

- [ ] 删除无执行意义的情绪形容词。
- [ ] 先写任务边界，再写输出格式，再写降档规则。
- [ ] 指定“只有上一级不足时才降档”。
- [ ] 要求对每一级空白显式声明，不允许跳过不报。
- [ ] 不让检索提示词直接生成最终定稿。
- [ ] 对正式成稿另开 writing 阶段，避免检索与执笔混线。

## 10. 常见失误

- 误把“多个角色提示词”当成“多个原生子代理”。
- 并行了需要共享判断标准的两个阶段，导致口径冲突。
- 检索阶段直接写成结论稿，后面无法追溯证据。
- bridge 阶段把间接证据说成直接证明。
- 没有显式保留空白和缺口，导致写作阶段过度外推。
