# 工程申报修订经验

## Purpose

这份参考文件沉淀老师批注驱动的工程申报补写经验，用于处理“已有正文 + 老师意见 + 逐条补写”的修订场景。

它不保存某个项目的具体事实，只保留可复用的方法、判断规则和措辞清理方向。

## 适用场景

- 老师在 `Word` 中标黄、批注或直接写出修改意见
- 用户希望先提取修改点，再逐条补写
- 新增内容需要并回现有申报书，而不是另起炉灶重写

## 批注驱动补写原则

1. 先保留老师原文，不先改写老师批注。
2. 先提取“原文 + 修改点”，形成逐条对照稿。
3. 补写内容只能增补，不能比老师给出的信息更少。
4. 先在对照稿中完成口径确认，再正式并回正文。
5. 用户未确认前，不直接在 `Word` 正文里大范围改写。

## 章节补写链条

面对某一技术方向的新增内容时，优先沿同一条逻辑链向后展开：

1. 技术背景
2. 现状概述
3. 存在问题
4. 原因分析
5. 技术清单
6. 价值与成熟度评估
7. 重点发展方向

不要只在其中某一节单独补一句，导致前后逻辑断裂。

## 现状概述写法

- 不只写结论，还要补形成现状的原因。
- 对比维度优先写：
  - 频谱
  - 速率
  - 功耗
  - 跟瞄
  - 成本
- 写高频通信时，可采用“微波单列，高频合写，太赫兹单独落一句”的写法。
- 避免把毫米波和太赫兹在频段上机械切开，导致重点发散。

## 存在问题写法

- 不写空泛问题，要拆成 3 到 4 个可落地的具体不足。
- 问题优先落到链路动作和系统能力。
- 常用问题维度：
  - 传输损耗
  - 波束捕获与跟踪保持
  - 高通量收发与实时处理
  - 多节点协同和后续链路运行衔接

## 原因分析写法

- 原因要和新增问题一一对应。
- 优先从下面几层写：
  - 传播机理
  - 器件工艺
  - 系统验证条件
  - 跨平台协同验证不足
- 不把原因分析写成解决方案广告。

## 成熟度评估写法

先写已完成的验证事实，再写已具备的应用基础，最后写仍需补充的试验和工程化验证。

建议顺序：

1. 已完成的验证或联合试验
2. 已具备的能力基础
3. 仍需补充的场景验证
4. 仍需补充的工程化验证

避免口语化收束，例如：

- 从单项验证走向体系应用
- 工程推广节奏
- 较稳

## 工程化措辞清理

### 避免解释腔

- 不在于……而在于……
- 造成上述问题的首要原因，在于……
- 这也是……的主要原因之一

### 避免口语化

- 也就是说
- 并不能
- 往往
- 尽快
- 及时
- 较稳

### 避免软件工程词

- 接口
- 闭环
- 压测
- 兜底
- 下游

### 避免生造词和动作状态混写

- 持续稳定建链
- 轮换接续
- 安全抗扰
- 异常扰动

## 替换方向示例

- `持续稳定建链` -> `快速建链、链路保持和持续通信`
- `安全抗扰` -> `抗干扰` 或 `通信安全、抗干扰`
- `下游分发衔接` -> `后续空中分发链路协同运行`
- `容错恢复` -> `故障恢复`
- `异常情况感知` -> `识别异常情况`

## 使用提醒

- 这份文件只提供补写方法，不替代具体项目事实核对。
- 需要写真实试验、联合验证、活动验证时，先核准事实，再并入正文。
- 如果用户任务重点是“怎么协作、怎么提问、怎么回填 `Word`”，转回上层协作编排 skill 处理。
