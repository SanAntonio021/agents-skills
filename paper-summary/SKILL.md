# 单篇论文总结

## 目标

只做一件事：读论文，并生成中文总结。

默认输入是本地 PDF、已下载论文、论文正文节选或已经确认可读的全文。缺少 PDF 或正式全文时，先调用 `paper-download`。

## 什么时候用

- 用户要求“总结这篇”“读这一篇”“整理这篇”“做文献卡片”时，只处理单篇论文，不扩成多篇综述。
- 用户只给 DOI、题名或出版社页，并且本地没有 PDF 时，先转 `paper-download`。
- 用户要求“下载并总结”时，先 `paper-download`，再本技能。
- 用户还在问“某个领域有哪些论文”“近几年代表作是什么”“帮我筛方向”时，先走 `research-lookup`。

## 输出

`paper-summaries/<year>-<journal>-<title-short>.md`

如果用户已有固定论文目录，默认把总结写到该目录根下的 `paper-summaries`。

## 保存到哪里

工作区根目录按下面顺序判断：

1. 当前任务正在处理的工作区根目录。
2. 当前 Git 仓库根目录。
3. 当前工作目录。

默认文件名：

`<year>-<journal>-<title-short>.md`

处理规则：

- `year` 优先取正式发表年份；拿不到就用 `unknown-year`。
- `journal` 优先取期刊名、会议名或论文来源的简短写法；拿不到就用 `unknown-journal`。
- `title-short` 用论文标题的简短版本，并清理成 Windows 可用文件名。
- 如果用户明确指定文件名，按用户指定。

## 怎么读论文

1. 优先从论文正文、首页、摘要、作者信息和图表中提取事实。
2. 读取 PDF 时先用内置 PDF 读取能力。失败后用 Python 库兜底，优先尝试 `pdfplumber`、`PyPDF2`、`fitz`。Windows 控制台如果出现 GBK 编码错误，改用 UTF-8 输出，例如设置 `PYTHONIOENCODING=utf-8`。
3. 不要一开始把全文全部塞进上下文。先提取首页、摘要、作者单位、DOI/ISSN、图表标题、结论和局限附近内容；需要补证据时再扩大范围。
4. 基础元信息优先从论文原文提取，顺序为：题目、作者、第一作者、通讯作者、单位、期刊名、卷期页码、发表/在线发表时间、DOI、ISSN。
5. DOI 页面或 Crossref 只能辅助校验，不能覆盖论文正文中已经明确的信息。
6. `中科院期刊分区`、`影响因子` 不要凭记忆填写，也不要只凭搜索引擎摘要填写。
7. 正式期刊论文要检查是否有 `Supplementary Information`、`Extended Data`、`Source Data`、`Supplementary Video` 等附件。优先从论文官网、出版社页、PDF 首页、Data availability 和 Supporting information 链接确认。与主结论直接相关的附件证据要纳入总结；附件不存在、本地未下载或无法访问时，在总结中写明“未纳入原因”，不要静默漏掉。
8. 如果用户需要图表信息，或论文总结要保留关键图，优先调用 `scripts/extract_paper_images.py` 提取图片。默认命令：
   `python scripts/extract_paper_images.py --pdf <PDF_PATH> --out <OUTPUT_IMAGE_DIR> --mode auto`
9. 图片提取优先级是：论文官网/出版社页/会议页等官方 HTML 原图，其次是脚本从 PDF 中按 caption 和 bbox 自动裁出的 `figures/` 图片，最后才用 `debug_pages/` 整页渲染排查。`debug_pages/` 只用于人工复核，不能自动放进 Markdown 正文。
10. `manifest.json`（`schema_version: 2`）中每张图会记录 `figure_id`、`caption`、`page`、`bbox`、`source_type`、`confidence`（`”high”/”medium”/”low”`）、`confidence_score`（原始数值）和 `warning`。写总结时：`confidence == “high”` 可进入正文，仍需快速目检；`confidence == “medium”` 人工目检通过后可进入正文；`confidence == “low”` 不进入正文，仅用于排查。低置信图片写”需人工复核”，不要静默错配。
11. `figures/` 里的 PNG 默认只保留图或表本体，不强求把论文原始 caption 一起截进图片；caption 以 `manifest.json` 保留，供回查和匹配使用。
12. Markdown 正文不要复制原文图注题名，也不要写“原文图注题名/原文表题”。图片 alt 文本只写编号，例如 `![Fig. 1](...)`、`![Table I](...)`；图片后只写一行中文导读，例如 `Fig. 1：说明这张图该看什么。`
13. 图件抽取后必须快速目检关键图，确认左/右轴标题、刻度、图例、底部横轴标题和 `(a)/(b)` 子图编号没有被裁掉；如果边缘文字缺失，先修 bbox 或重新提取，不要把缺边图放进总结。
14. 图和表都按原论文编号做一次顺序检查。正文里出现 `Fig. 1`、`Fig. 2` 后直接跳到 `Fig. 7`，必须说明 `Fig. 3-6` 为什么没放；不能让读者以为是漏抽或漏写。
15. 表格和图片同等重要。只要表格支撑主要结论、实验条件、性能对比或关键参数，就应放进 Markdown 正文；只有裁剪置信度低、内容明显错配或需要人工确认时，才放入“需复核”说明。
16. 原文长图注和表题不要丢。正文只写编号和中文导读，完整原文 caption 保存在 `manifest.json`，以后需要核对时再回查。

## 图表与附件写入规则

- 正文优先放能帮助读者理解论文主线的图表：系统框图、实验平台、关键结果曲线、性能对比表、参数表、消融或补充验证。
- 图表编号沿用原文：`Fig. 1`、`Fig. 2`、`Table I`、`Supplementary Fig. 7`、`Extended Data Table 1`。不要自己改成“图一”“表二”，也不要重排编号。
- 每张图或表后只写一句中文导读，说明读这张图要看什么。不要在正文里加“原文图注题名”“原文表题”这类提示词。
- 如果使用附件图表，单独放到 `## 补充材料与扩展数据`。只整理和主结论有关的证据，不全文翻译附件。
- `manifest.json` 的 `skipped`、`warning` 和 `sequence_audit` 要检查。发现低置信图、缺号或被跳过的图表时，先判断是否需要补抽；不补抽就写清原因。
- `debug_pages/`、`embedded_candidates/` 和整页渲染图只用于排查。Markdown 正文不要引用这些文件。

## LetPub 查询

正式期刊论文默认用 `web-access` 进入 LetPub 期刊查询页做站内检索，速度慢可以接受，稳定和可核验优先：

`https://www.letpub.com.cn/index.php?page=journalapp`

查询顺序：

1. 先加载 `web-access` skill，并按其要求检查 CDP/浏览器代理可用性。
2. 打开 LetPub `journalapp` 页面。
3. 优先在页面表单的 ISSN 字段中填写完整 ISSN，例如 `2095-8099`，提交查询。
4. 如果没有 ISSN、ISSN 无匹配，或结果与论文期刊明显不一致，再用期刊名精确查询。
5. 只有页面返回类似“搜索条件匹配：N条记录”且结果表中出现可核验字段时，才算确认。

优先记录：

- ISSN
- LetPub 页面展示的期刊名，包括别名或缩写
- 影响因子
- 最新分区信息，优先写 LetPub 页面显示的分区口径
- 大类学科和小类学科
- SCI/SCIE/SSCI 等收录状态

LetPub 失败状态要写清楚原因，例如：

- `待查询（LetPub 页面无法打开）`
- `待查询（LetPub 站内检索无精确匹配）`
- `待查询（LetPub 结果与论文期刊信息冲突）`
- `待查询（疑似登录墙或访问拦截，未能读取结果表）`

如果当前论文没有正式期刊来源，`中科院期刊分区` 和 `影响因子` 写 `不适用` 或 `待查询`，不要硬查、不要猜。

## 怎么写

- 全文输出中文。
- 论文标题、期刊名、作者名、DOI、URL 可保留原文。
- `关键信息` 使用 Markdown 表格。
- 正文主体使用 `##` 作为大块分组标题。
- 每个具体点使用 `###`，不要继续写成一长串横杠列表。
- 不写主观评价，不做推荐结论。
- 不做多轮“概述 + 详细总结”重复展开，只保留一个展开版本。
- `术语与专名` 放在前部，紧跟 `关键信息` 之后；它是阅读辅助，不放到文末。优先收录标题、摘要、图表和一句话总结里会遇到的缩写、器件名、算法名和关键指标。
- 各节按模板注释分工写作。研究问题、研究方法、实验结果、结论与局限之间不要互相复述。
- 关键数据优先用表格呈现，尤其是频率、距离、速率、调制格式、EVM、BER、SNR、带宽、器件增益、功耗、噪声系数等指标。
- 图表信息里的图片和表格截图必须能在 Markdown 预览中直接显示，写法用 `![Fig. 1](相对路径)` 或 `![Table I](相对路径)`，图片后再接一句中文导读。优先引用 `images/<paper-short-name>/figures/fig_1.png`、`images/<paper-short-name>/figures/table_1.png` 这类路径，不要引用 `debug_pages/`。
- 如果附件或 Extended Data 含有支撑主结论的图表，单独增加 `## 补充材料与扩展数据`；只整理关键证据，不全文翻译附件。优先放：关键 Supplementary Note 结论、Extended Data 对比表、补充实验设置、功耗/复杂度/稳定性/距离扩展等限制信息。
- 如果 `manifest.json` 只有 `debug_pages/` 或图片置信度低于阈值，正文不要硬塞整页截图；改写为“该图需人工复核或从官网原图补齐”。
- 样本阶段或用户要求跳过 LetPub 时，`中科院期刊分区` 和 `影响因子` 统一写 `待查询（样本阶段跳过 LetPub）`，不要空着或猜测。
- 引用原文时，只摘最关键的少量原句或原始结果，不长段抄录。
- 无法确认的信息写 `未说明`、`待查询` 或 `不适用`，不要编造。

## 边界

- PDF 下载、正式版判断、下载索引维护：用 `paper-download`。
- 多篇文献综述、代表作筛选：先用 `research-lookup`。
- 投稿文章、引言综述、申报书材料：转 `sci-writing` 或 `project-writing`。
- 批判性评审、同行评议意见、审稿式打分：转 `paper-review`。

## 会用到的工具

- PDF 获取：`paper-download`
- LetPub 和网页访问：`web-access`
- PDF 后处理：`pdf`
- 图表提取：`scripts/extract_paper_images.py`
- 学术写作总控：`sci-writing`
- 文稿审查：`paper-review`

## 以后怎么维护

- 如果以后调整总结模板，优先只改 `## 总结模板`。
- 如果需要新增固定输出字段，先确认它是原文可提取事实，不要把推断意见塞进模板。
