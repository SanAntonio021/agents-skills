# API 中转站计费审计

## 目标

回答三个问题：

1. 某段时间实际扣了多少，能否由请求记录解释。
2. 页面标注倍率是否真正作用于当时请求。
3. 在相同模型、Token 负载和充值口径下，哪个站的人民币成本最低。

不要把“当前显示倍率低”直接等同于“历史消费低”或“人民币实付低”。

## 硬边界

- 只审计用户有权访问的账户和数据。
- 联网前先加载并遵守当前可用的 `web-access` skill，优先后台请求和浏览器网络记录。
- 默认只读。不要充值、退款、改分组、改模型、创建或删除密钥。
- 付费探针会产生真实消费；只有用户明确同意本次测试及费用上限后才能执行。
- 不在对话、日志、截图或文件中输出 API key、Cookie、JWT、Authorization header、完整支付单号或带令牌 URL。统一写成 `[REDACTED]`。
- 不能证明的历史倍率、站内单位或扣费公式写 `not_evaluable`，不要拿当前页面值回填历史。

## 开始前

先从当前对话提取已有信息，避免让用户重复。至少确认：

- 站点及账户范围
- 审计时间段和时区
- 模型名或模型别名
- 要查单站异常，还是比较多个站
- 人民币成本采用哪批充值记录

如果缺少的信息会改变结论，只问一个最关键问题。已有登录会话时，使用现有会话；不要让用户在聊天中发送密码或密钥。

## 证据等级

按下面顺序使用证据：

1. `E1 请求级证据`：原始 usage/log 响应中的 Token、请求时间、扣费、请求级倍率和 request ID。
2. `E2 账本证据`：余额变化、消费流水、充值到账、赠送、退款和调整。
3. `E3 配置证据`：带采集时间的当前模型价格、账户分组和倍率配置。
4. `E4 展示证据`：控制台卡片、说明文字或营销页。

历史结论优先依赖 E1/E2。E3/E4 只能证明采集时刻，除非站点明确返回带生效时间的历史配置。

## 工作流

### 1. 建立证据清单

记录每个站点的：

- 证据来源、采集时间、时区和覆盖时间段
- 页面显示单位，例如 CNY、USD、credit、quota
- 当前分组名和当前倍率
- 是否能取得请求级历史倍率
- 是否能取得充值与余额流水

先标出缺口，再计算。不要用猜测补齐关键字段。

### 2. 采集当前价格与账户配置

读取公开价格页和登录后的账户配置。保存模型完整名称、输入/缓存/输出价格、价格单位、分组名、倍率及采集时间。

模型别名相同不代表上游模型、快照或服务质量相同。只把价格可比性写成价格结论，不扩展成质量结论。

### 3. 采集历史请求与账本

按目标时间段拉取全部分页，保留原始精度。优先提取：

```text
station, request_id, created_at, timezone, model,
input_tokens_total, uncached_input_tokens, cache_read_tokens,
cache_write_tokens, output_tokens, request_count,
base_input_price, base_cache_read_price, base_cache_write_price,
base_output_price, model_ratio, group_ratio, cache_ratio,
output_ratio, fixed_fee, observed_debit, debit_unit,
opening_balance, closing_balance, recharge_credit, paid_cny,
evidence_level, source_timestamp
```

字段缺失就留空并说明，不把站点返回的总输入 Token 自动当成未缓存输入 Token。

门户接口发现、分页、动态倍率和安全处理见 [references/portal-api-patterns.md](references/portal-api-patterns.md)。

### 4. 识别计费方案

先判断站点属于哪一种，再套公式：

- 按公开模型单价计费，再乘分组倍率
- 按加权 Token、模型倍率和分组倍率折算 quota
- 日志已经返回最终货币/额度扣费
- 固定请求价或混合计费

模型基础价格和 `model_ratio` 往往是两种表达方式。除非站点公式明确要求，不要同时使用，避免重复计费。完整公式见 [references/billing-formulas.md](references/billing-formulas.md)。

### 5. 分别计算预期值、观察值和人民币实付

每笔请求至少保留三列：

- `expected_station_debit`：由当时可证明的价格和倍率算出的站内扣费
- `observed_station_debit`：日志或账本实际扣费
- `real_cny_cost`：结合实际充值到账换算的人民币成本

然后计算差额和相对误差。显示时可以四舍五入，核算过程保留原始精度。

### 6. 做余额对账

用期初余额、非消费入账、非消费出账和期末余额反推总消费，再与请求日志合计比较。

重点排查：漏页、时区边界、重复记录、失败请求扣费、退款、人工调整、赠余额、多个密钥共用账户，以及日志保留期不足。

### 7. 比较多个站

只在以下条件一致时排名：

- 同一模型或明确可比的模型快照
- 同一组输入、缓存和输出 Token
- 同一计费时间点
- 各站采用各自真实充值成本
- 站内 credit/quota 已换成同一人民币口径

同时给出：

- 当前标价排名
- 按实际充值换算后的人民币排名
- 历史区间实付排名

数据不足时只给可计算部分，不强行选“最便宜”。

### 8. 必要时做最小付费探针

只在只读证据不足且用户已明确批准时执行：

1. 先记录余额和当前配置。
2. 固定模型、短输入和最大输出 Token。
3. 每站只发满足验证所需的最少请求。
4. 记录 request ID、Token usage、请求前后余额和新生成日志。
5. 立即停止，不自动扩大样本。

探针只能证明测试时刻，不能反推历史倍率。

## 常见误判

- 当前 `0.08x` 不等于昨天所有请求都是 `0.08x`。
- 输出 Token 通常比输入 Token 贵；只看总 Token 数会低估成本。
- 总输入 Token 可能已经包含缓存命中 Token；重复相加会高估。
- `quota`、`credit`、USD 和 CNY 不是天然等价单位。
- 充值折扣、赠送和手续费会改变人民币成本，但不会改变站内日志扣费。
- 页面展示的“总消费”可能包含其他密钥、模型或时间段。
- 请求日志合计与余额减少不一致时，先对账，不先认定站点多扣。

## 输出格式

按此顺序输出；无数据的部分保留并标 `not_evaluable`：

```text
结论
- 单站异常是否能解释
- 当前最便宜与历史最便宜

核算口径
- 时间段、时区、模型、Token 口径、充值口径

证据
- E1/E2/E3/E4 来源及采集时间

计费明细
| 站点 | 基础成本 | 请求级倍率 | 预期扣费 | 实际扣费 | 人民币实付 | 差额 |

余额对账
- 账本反推消费、日志合计、未解释差额

价格比较
- 当前标价排名
- 人民币实付排名

无法判断
- not_evaluable 项及缺少的证据
```

结论必须区分“已证明”“高度可能”“无法判断”。不要用小数点后的微小差异掩盖证据缺口或模型质量差异。

## 维护

- 站点接口和字段会变化。把具体 URL 当候选模式，不当固定协议。
- 新发现的通用计费公式放入 `references/billing-formulas.md`。
- 新发现的通用门户字段和安全读取方式放入 `references/portal-api-patterns.md`。
- 不把某个用户的域名、账户、密钥、充值记录或消费明细写进 skill。
