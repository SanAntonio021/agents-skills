# 跨格式导出与排版降噪最佳实践

本协议总结自跨格式数据流转（如 Markdown 导出至 Excel、Word 表格、或非富文本编辑器）时的强规则踩坑教训。

## 核心教训

### 1. Markdown 标记符（如 `**`、`#`）在非原生平台中会暴露为乱码或脏数据

**问题**：大模型习惯使用 Markdown 原生语法（如 `**加粗**`）进行逻辑分层或强调。但直接用代码（如 Python `openpyxl`）提取文本并回填到 Excel、传统 Word 文档或不兼容 Markdown 的系统中时，这些符号会作为原生字符串原样显示，导致正式提交的交付件中存在极不专业的“排版脏数据”。

**对策**：

- **拦截检查 (Pre-flight Check)**：在执行“代码提纯导出至 Excel/Word”的动作之前，**必须前置执行自动“Markdown 语法清洗”**。
- **代码级降阶处理 (Data Sanitization)**：利用正则或文本替换，剥离原文本中的 `**文本**`、`# 标题` 等符号，仅保留净文本。如果目标软件（如 Excel）确实需要加粗样式，【强制】在执行代码（Python）层面控制其样式特征，取代直接将 `**` 作为文本字符串填入单元格的做法。
- **全域排雷 (Final Audit)**：在生成最终非 Markdown 格式文档前，进行一次底层文法排雷，专门匹配并拦截遗落的富文本标记字符，确保数据绝对整洁。
