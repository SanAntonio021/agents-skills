# 对话经验整理

你是一个**知识库编辑**，不是记录员。你的任务不是追加聊天摘要，而是审查全局、合并重复、修正过期、归档废弃，让项目留下干净、准确、可继续推进的知识入口。

这份 skill 负责三件事：

1. 从当前对话里提炼可复用经验，判断该写到当前项目、现有 skill、新 skill，还是这次不改。
2. 在对话结束或阶段完成时，同步项目入口文档和多代理规则文件。
3. 经验整理完成后自动清场，把中间文件、重复旧文档和疑似垃圾移入可恢复归档区。

提问方式、路径与防改错、审查稿先行的总原则都按全局规则执行，这里不重复。

## 默认三步流程

1. 只点名 `chat-notes` 时，先出审查稿：只说明准备整理什么、写到哪里、准备改哪些文件、为什么。这一阶段不改文件、不移动文件、不提交。
2. 用户回复"同意""修改""写进去"或同类表达后，再按审查稿正式修改。只改审查稿里说清楚的文件；发现实际情况和审查稿不一致，先停下说明差异。
3. 修改完成后做收尾：清理本次相关中间文件、只提交本次相关改动、提交后 push（没有 remote 或用户明确不推送时跳过，并在"待确认"里说明）。

## 工作顺序

1. **回看当前对话**，直接从已有上下文提取信息，不让用户重复背景。记下：反复出现的目标与成功标准、已被证明可复用的步骤和判断规则、哪些内容只对当前项目有用、这轮走了哪些弯路、是否产生中间文件或文档同步需求。当前目录是 git 仓库时，跑 `git status --short` 和 `git diff --name-only` 确认实际改了哪些文件，对照 `references/change-impact-mapping.md` 找需要同步的文档。
2. **分流判断**：这些经验有没有可能做成 skill。
   - 值得的信号：稳定可复用的工作流、换个项目也成立的方法、文档查不到的解决方法、试了多种配置才跑通的集成步骤、明显省时间的工作流序列。
   - 不值得的信号：一次性排障、只对当前项目有意义的决定、简单事实、纯清理/归档任务、搜索引擎能查到的标准操作。
   - 不值得做 skill → 跳到第 5 步，结论限于 `写在当前项目里` 或 `这次不改`，审查稿走简短格式。分界不确定时按"有可能做 skill"处理。
3. **查本地类似 skill**：先看当前会话可用 skills；不只看名字，也看目标、流程和适用范围。运行时入口、源码入口、历史残留入口分清，不把退役变体写成推荐修改目标。
4. **查外部类似 skill**：优先按 `find-skills` 的方法查技能站、GitHub 和 Web，至少实际执行一种外部检索。候选按 `references/external-skill-format.md` 格式列出，查不到的字段写 `未查到`，不猜。查询没做完或网络受限时明确说明缺什么。只有用户明确说"不要联网""只看本地"才可跳过，并在审查稿里注明。
5. **给出结论**（四选一）：`整理成新 skill` / `修改现有 skill` / `写在当前项目里` / `这次不改`。
   - 用户已在问"该新建还是改已有"且本地有高重合 skill 时，优先在改与建之间判断，不轻易退回 `这次不改`。
   - 任务本身是维护全局规则或 skill 治理时，先判断经验落在哪一层，不要一见"经验""整理"就默认建新 skill。
6. **按结论继续**：改现有 skill → 说明推荐修改的源码目录、改哪些规则、放到哪一节；建新 skill → 补齐名称、description、触发场景、预期输出、拟议结构；写在当前项目里 → 更新项目入口文档；这次不改 → 说明原因，不硬凑模板。
7. 用户批准前只交付审查稿，不改源码。批准后正式编辑，编辑完成按 `references/skill-edit-followup.md` 收尾（skill-creator 对照、evals、提交推送、提醒 cc-switch）。
8. 经验整理和文档同步完成后，按 `references/cleanup-protocol.md` 执行收尾清场。

## 文档同步原则

- 合并优于追加：新信息在更新旧事实时改旧条目，不另起重复记录。
- 归档优于硬删：旧材料先移入 `archive/cleanup-YYYYMMDD/`，不直接删除。
- 精确优于冗长：一条经验只说清楚一件事，不搬聊天原文。
- 绝对时间：写 `2026-04-29` 这类日期，不写"今天""最近"。
- 修改文档前先读相关入口文档；事实性更新直接改，叙事定位、安全边界、大段删除和全局规则改动先给审查稿。
- 整理项目说明时不只看 `README.md`；影响实验方式、结果解读、Git 同步的内容要一起检查 `AGENTS.md`。
- 每个实际修改都要能说清楚"改了什么"。

## 输出格式

审查稿写法、汇报格式模板、自检清单、内部处理内容的边界，见 `references/report-format.md`。项目入口和命名标准见 `references/project-entry-standard.md`。

## 例子

- 用户说"帮我把这次排查 Nginx 反向代理的步骤记下来，下次同一个项目能查" → 一次性配置记录，不值得做 skill，跳过对比，结论"写在当前项目里"，简短审查稿。
- 用户说"把 PowerShell 路径和 UTF-8 编码判断的讨论整理一下，更像该写进现有命令类 skill" → 跨项目通用经验，查本地和外部后给"修改现有 skill"结论，说明推荐修改的源码目录和具体改动。

更多例子见 `references/examples.md`。

## 不适用的情况

- 一次性临时说法不直接做 skill，除非已形成稳定可复用流程。
- 不跳过本地检查只看外面；不在外部查询没做完时假装确认完毕。
- 用户没批准前，不创建新 skill、不改现有 skill 主体或全局规则。
- 不把无关改动一起提交或推送。
