# 审查稿与汇报格式

## 审查稿写法

审查稿默认给用户看，不是给另一个 agent 看。

- 先说结论，再说理由。
- 用短句，不堆抽象词。
- 少用"沉淀、分层、入口、链路、治理、规范化"这类词；必须用时，后面补一句白话解释。
- 不写大段过程复盘，只写这次要不要改、准备改哪里、准备整理什么、为什么。
- 审查稿只列准备整理或准备修改的内容。不要列"不整理""不处理""不会去动"的清单；这些只作为内部过滤依据。
- 如果需要说明为什么某些内容不写进去，只写一句短理由，例如"其他材料与本次准备写入内容无关，暂不进入审查稿"。不要逐条列出被排除的文件、话题或旧讨论。
- 每条建议都要能直接回答"我接下来要不要让你写进去"。
- 如果审查稿或准备写进文件的正文里，有用户刚表示看不懂的词，换成更具体的话。
- 分流后不做 skill 时，审查稿只需写明结论、理由、写到哪里和内容摘要，不需要 skill 对比。

## 汇报格式模板

默认按这个结构汇报，从上到下读一遍就够，没发生的部分直接省略：

```text
结论：整理成新 skill | 修改现有 skill | 写在当前项目里 | 这次不改

理由：
- <reason 1>
- <reason 2>

准备整理：
- <要写入或同步的内容 1>
- <要写入或同步的内容 2>

本地有没有类似 skill：
- <existing skill + difference>

外面有没有类似 skill：
| skill | 来源 | 链接 | GitHub stars | 下载/安装指标 | 匹配度 | 结论 |
| --- | --- | --- | --- | --- | --- | --- |
| <external skill> | <source> | <url> | <stars/未查到> | <指标/未提供> | <高/中/低> | <结论> |

实际变更：
- <file>：<具体改了什么>

归档清理：
- <archive path>：移动了 <N> 个文件，清单在 <manifest>

待确认：
- <只放需要用户拍板的选择，没有就整节省略>
```

- "待确认"只能放需要用户做决定的事项。不要放无关的旧背景、上一轮已讲清的状态、本任务范围外的事、或 AI 自己的操作说明。没有真正待用户拍板的事，就把整节删掉，不要写"无"也不要硬凑。
- 分流后不做 skill 时，省略"本地有没有类似 skill"和"外面有没有类似 skill"。
- 外部 skill 候选的字段规则见 [external-skill-format.md](external-skill-format.md)。

## 不放进审查稿、由 AI 内部处理的内容

- **踩坑记录**：这轮遇到的操作层面错误和避免方法，写进 memory，不给用户看。
- **Git 状态详情**：正常时在末尾写"已提交并推送"即可；只有出问题（未提交、混入无关文件、push 失败）时才展开说明。
- **排除清单**：不准备整理的话题、文件、旧讨论和临时输出，只在内部用于避免误写，不向用户逐条列出。

## 自检清单

完成后过一遍：

- [ ] 结论属于四类之一，且理由清楚
- [ ] 分流判断有依据，适用范围不确定时走了完整流程
- [ ] （分流为"有可能做 skill"时）已搜索本地和外部类似 skill，外部候选按格式列清楚且缺失字段没有编造；或已说明按用户/网络限制跳过
- [ ] 源码目录和同步产物没有混写，README/AGENTS/CLAUDE 职责没有混乱
- [ ] 规则文件保护到位，未把 `AGENTS.md`、`CLAUDE.md`、`GEMINI.md` 当垃圾
- [ ] 项目会同时用 Codex 和 Claude 时，已同时检查 `AGENTS.md` 和 `CLAUDE.md`；没有明确 Claude 使用场景时，未主动新建 `CLAUDE.md`
- [ ] 没有相对时间遗留
- [ ] 清场已生成清理清单，或明确说明没有可清理项
- [ ] 最终汇报写明实际变更、归档清理、Git 推送状态；"待确认"只在有需要用户拍板的事项时才出现
