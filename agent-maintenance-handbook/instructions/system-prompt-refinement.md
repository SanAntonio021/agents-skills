# 系统提示词和主规则精简

## 这份说明管什么

这份说明只管一件事：底层规则怎么改，才不会把一次性问题写成长期规则。

## 这里管什么

这里讨论的是：

- 智能体底层约束
- 全局路由与维护规则
- 长期会反复用到的行为约定

## 这里不管什么

下面这些，不在这里展开：

- 普通任务输出风格
- 单项目事实
- 临时老师意见或一次性偏好
- 本应进入独立技能的业务流程

这份说明也不是具体模板写法手册。碰到本地系统提示词或其他具体提示词模板的写法问题，默认参考 `<agents-root>\prompt_guidelines\openai.md`。

## 基本原则

1. 只有在真实失败、真实冲突或真实重复模式出现后，才新增全局规则。
2. 规则必须尽量短、客观、可执行。
3. 能放进独立技能的，不放进全局规则。
4. 能放进项目目录的，不放进全局规则。
5. 新规则应尽量替换旧规则，而不是不断叠加。

## 该写到哪里

决定“这条内容该写到哪里”时，按这个顺序判断：

- 跨工作区、跨新对话都要生效的机器级路由与维护约定：优先写入 `%USERPROFILE%\.codex\AGENTS.md`，必要时再同步到 `AGENTS.md`
- 当前同步技能仓里的查找顺序、目录模型和维护约定：写入 `AGENTS.md` 或对应说明文档
- 可复用任务流程：写入对应技能
- 仅用于维护智能体本身的方法：写入 `agent-maintenance-handbook/instructions/`
- 单项目限定规则：写入项目目录

## 修改顺序

1. 先指出真实触发问题是什么。
2. 再判断它属于：
   - 全局规则问题
   - 技能查找顺序问题
   - 某个具体技能正文的问题
   - 某个项目本地问题
3. 只有确属全局层时，才改底层规则。
4. 改完后检查是否有旧规则已被覆盖，应一并删去。

## 写具体提示词模板时怎么分工

当任务已经进入“写或改具体提示词模板”这一步时，按下面的分工处理：

- 本文件负责判断这条内容该不该进入底层规则，以及该写在哪一层。
- 具体提示词模板的结构化写法，默认参考 `<agents-root>\prompt_guidelines\openai.md`。
- 优先落实其中与当前任务直接相关的 `Context / Task / Constraints / Output`、作用域纪律、约束精确化和缓存布局原则。
- 不把 `openai.md` 里的整套通用写法原样抄回全局规则；这里只保留经过本地验证后仍需长期复用的维护规则。

## 什么样的规则值得留下

优先保留这种规则：

- 明确触发条件
- 明确允许或禁止的动作
- 尽量能映射到工具或文件层行为

尽量别留下这种规则：

- 纯态度词
- 纯审美词
- 没有边界的“大而全”原则
- 依赖一次性上下文才能理解的补丁句子

## 主规则怎么做减法

处理 `AGENTS.md`、`CLAUDE.md`、`GEMINI.md` 这类主规则文件时，先做减法。

先把内容分成三类：

1. 纯重复
2. 本地覆盖
3. 只在当前本地环境成立的长期约定

处理顺序：

- 纯重复先删，不要换句话再写一遍
- 本地覆盖可以保留，但要直接写出“本地差异”本身
- 只有离开当前本地环境就不成立的内容，才写成本地长期约定

主规则里优先只留这些内容：

- 不写进去就容易出错的硬规则
- 本地明确覆盖上游默认行为的规则
- 只在当前目录结构、当前客户端环境里成立的长期约定

下面这些通常不要塞回主规则：

- 历史说明
- 过渡阶段才有用的目录说明
- 长别名表、长路由表
- 只是表达偏好或写作口味

不要整段照搬上游通用规则。只有本地确实还需要长期强调，或需要改成更贴当前环境的短版本时，才保留本地写法。

## 三份主规则什么时候一起改

改 `AGENTS.md`、`CLAUDE.md`、`GEMINI.md` 时，共通规则默认同步。

只有下面两种情况，才分开写：

- 平台差异
- 明确只在本地某一端成立的长期差异

## 维护时注意

- 每次改底层规则时，优先删旧规则，避免只增不减。
- 如果某条规则已经稳定外拆为独立技能，应把 handbook 留成索引和维护原则，不重复保留整套流程。
