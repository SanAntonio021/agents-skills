# 本地文稿版本保护

本规则只保护会被智能体实际写入的本地 `.md`、`.tex` 及其必要源文件。它保存可恢复的文件版本，不承担对话摘要、上下文压缩或云端上传。

## 何时执行

执行版本保护：

- 当前任务将修改工作区内的 `.md` 或 `.tex`；
- 修改同时涉及正文依赖的 `.bib`、模板、绘图脚本或必要源数据；
- 当前轮修改完成且验证通过，需要形成一个可恢复版本。
- 用户对刚完成的本地文稿明确说“这版可以”“结构定了”“投稿候选”或同义表达；
- 用户要求查找、比较或恢复本地文稿的旧版本。

不执行版本保护：

- 只读审查、问题诊断、文件内容提取；
- 只在聊天中改写句子，未写入本地文件；
- 只讨论方案，尚未修改文件；
- 普通 `Ctrl+S` 或用户自己保存文件；
- PowerPoint 手工图件。PPTX 版本由 `paper-figure-review` 的“留档当前页”加载项处理。

一次“修改轮”以用户的一次修改要求为单位。可以包含多个相关文件和多次内部保存；只有整轮完成并验证通过后才创建一个 commit，不按保存次数创建版本。

## 修改前检查

1. 锁定本轮目标文件和允许纳入的依赖文件，形成显式路径清单。
2. 查找现有仓库：
   - 对目标文件运行 `git -C <目录> rev-parse --show-toplevel`。
   - 找到仓库时先把返回的绝对根目录继续交给第 3 步校验；只有根目录范围合格时才沿用，不另建嵌套仓库。
   - 如果返回的是 `D:\BaiduSyncdisk`、`D:\BaiduSyncdisk\Paper` 等被误初始化的公共父目录，立即停止。提示用户先处理该父仓库；不能静默沿用，也不能自动在项目内再建嵌套仓库。
   - 没找到仓库时，候选根目录只能是当前工作区的准确项目根目录。
3. 初始化前显示候选根目录的完整绝对路径。以下位置不得初始化：
   - 盘符根目录、用户主目录、云盘根目录；
   - `D:\BaiduSyncdisk`、`D:\BaiduSyncdisk\Paper`、`D:\BaiduSyncdisk\申报书本子` 等容纳多个项目的公共目录；
   - 当前工作区明显包含多个无关项目的目录。
4. 候选根目录过大时停止。请用户改为打开具体项目文件夹，不能自行选一个更深目录猜测。
5. 没有 Git 时只问一次，并把准确路径和 baseline 范围放进同一个问题：

   > 将在 `<项目绝对路径>` 初始化本地 Git，并把 `<文件清单>` 当前状态保存为修改前 baseline。只做本地版本记录，不配置远程、不上传。是否执行？

6. 用户同意后在项目根目录执行 `git init -b main`。用户拒绝时，本轮继续按用户要求工作，但不初始化、不 commit，且本轮不重复询问。

如果已知百度云目录恢复成双向同步，立即停止在该目录创建或继续使用 `.git`。先提醒用户迁出仓库或恢复单向备份；不得依赖云盘同步 `.git`。

## Baseline 范围

默认只纳入：

- 目标 `.md`、`.tex` 正文；
- 正文引用的 `.bib`；
- 必要的 `.cls`、`.sty`、`.bst`、模板和配置；
- 可复现图件的 Python、MATLAB、TikZ、SVG、draw.io 等源文件；
- 本轮修改确实依赖、且体量合理的源数据；
- 为本仓库补充的 `.gitignore`。

默认不纳入：

- LaTeX 缓存和编译中间文件，如 `.aux`、`.log`、`.fls`、`.fdb_latexmk`、`.synctex.gz`、`.toc`、`.out`、`.blg`；
- 编辑器临时文件、备份文件、`__pycache__`、虚拟环境；
- 可重新生成的草稿 PDF/PNG/TIFF 和批量中间导出；
- 大型源数据、下载文献库；
- `.pptx`、`.pptm`、`.ppam`。大型 PowerPoint 文件继续由百度云单向备份。

不要把 `*.pdf`、`*.png` 一刀切写入 `.gitignore`，因为最终投稿件或选中的参考图可能需要长期保留。只显式纳入用户选中的 Image 2 参考图和最终投稿导出件。

新仓库需要 `.gitignore` 时，保留既有内容，只补缺失的缓存和临时文件规则。不得覆盖用户已有忽略规则。

默认补入以下 Office 图源规则，避免大型演示文稿持续污染状态：

```gitignore
*.pptx
*.pptm
*.ppam
~$*
```

确有必要跟踪的小型 PowerPoint 模板必须由用户明确指定，再作为单文件例外处理。可再生成的 PDF/PNG 不按扩展名全局忽略；只把项目中实际存在的中间输出目录或明确路径写入 `.gitignore`，例如 `figures/generated/`。最终投稿导出件和选中的参考图保持可显式纳入。

## 已有修改与 baseline

修改目标文件前，记录：

```powershell
git status --short
git diff -- <目标文件清单>
git diff --cached -- <目标文件清单>
```

- 目标文件干净：直接进入本轮修改。
- 目标文件已有未提交内容：先问是否把当前状态保存为 baseline。
- 新仓库中的既有目标文件视为待保存 baseline；初始化许可和 baseline 许可合并为一个问题。
- 用户同意：只提交目标稿及必要依赖，说明使用 `文稿：<文件或章节>｜修改前基线`。
- 用户拒绝：不得把旧改动伪装成智能体本轮改动。若无法可靠分离旧改动和新改动，本轮不创建正常 commit，并在交付时说明。

无关文件无论已修改、未跟踪还是已暂存，都不纳入 baseline，不要求用户清理，也不改变其状态。

## 每轮提交

修改完成后先验证，再提交：

1. 检查目标 diff，确认没有误改事实、数字、引用、LaTeX 命令或任务外文件。
2. 运行与任务相称的验证。对已跟踪文件同时执行 `git diff --check -- <本轮文件清单>` 和 `git diff --cached --check -- <本轮文件清单>`，覆盖 working tree 与 index。新文件先用 `git add -- <本轮新文件清单>` 显式暂存，再执行 cached 检查；不能让无关脏文件的空白问题阻断本轮提交。LaTeX 工程还要编译或运行当前项目已有检查；脚本图源要运行对应测试或最小重绘验证。
3. 验证通过且存在实际 diff 时，创建一个本地 commit。
4. commit 格式固定为：

   ```text
   文稿：<文件或章节>｜<修改目的>
   ```

5. 没有实际 diff 时不创建空 commit。

隔离规则：

- 只能使用显式路径清单；禁止 `git add .`、`git add -A`、`git commit -am`。
- 提交前核对 `git diff --cached --name-only` 和原始 `git status --short`。
- 若索引中已有无关暂存内容，不能用普通 `git commit` 把它们带入。对已跟踪目标文件使用 `git commit --only -- <本轮文件清单>`，并确认无关暂存状态保持不变。
- 相关新文件只用 `git add -- <本轮新文件清单>`，完成 `git diff --cached --check -- <本轮文件清单>` 后，同样用 `git commit --only -- <本轮文件清单>`。该命令会忽略清单外已暂存内容；提交前后都要核对无关 index，不能擅自取消暂存。
- commit 后用 `git show --name-status --stat HEAD` 核对只包含本轮文件。

本规则不执行 `git remote add`、`git push`、自动建远程仓库或上传。项目即使已有远程，也只创建本地 commit。

## 验证失败与 WIP

- 验证失败时不创建正常 commit，继续修复并重新验证。
- 因外部阻塞必须中止任务时，先说明失败项，再问是否保存 `wip`。
- 用户同意后只提交本轮相关文件，说明使用 `wip：<文件或章节>｜<阻塞原因>`。
- 用户不同意时保持工作区现状，不提交、不丢弃修改。

## 里程碑 tag

用户明确说“这版可以”“结构定了”“投稿候选”或同义表达时，在对应 commit 上创建 annotated tag。普通完成、普通保存或智能体自行判断不创建 tag。

- 当前轮有修改：先完成验证和 commit，再打 tag。
- 当前轮没有新修改：给最近的相关 commit 打 tag，不创建空 commit。
- 名称格式：`milestone/YYYYMMDD-HHmmss-<类型>`，例如 `milestone/20260712-153045-投稿候选`。秒级时间避免连续确认时重名；若仍冲突，在末尾追加 `-2`、`-3`。
- annotation 格式：`里程碑：<文件或章节>｜<用户确认语义>`。
- 创建后用 `git tag -n --list 'milestone/*'` 核对。

## 查找与恢复旧版本

用户要找旧版本时，先给候选，不直接改文件：

1. 按文件查看带日期和中文说明的历史：`git log --date=local --format='%h %ad %s' -- <文件>`。
2. 对最相关的 2-5 个候选显示提交说明、日期、涉及文件和摘要 diff。
3. 用户选定候选后，先确保当前目标状态已有 commit 或得到明确处理决定。
4. 只恢复用户选定的文件，验证 diff 后创建新的恢复 commit，例如 `文稿：<文件或章节>｜恢复至 <日期或版本>`。

不得使用 `git reset --hard`、`git checkout --` 或改写历史。恢复必须形成新 commit，使恢复动作本身也可撤销。
