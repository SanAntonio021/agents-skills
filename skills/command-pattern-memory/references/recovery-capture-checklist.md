# Recovery Capture Checklist

## Purpose

把“本线程内先失败、后成功”的高风险 Windows 命令，尽快收口成一次最小可复用的模式库更新，避免只在当前对话里临时修好。

## Trigger

满足以下条件时进入本清单：

- 当前线程里某条 Windows / PowerShell / 外部 CLI 命令已经失败过
- 后续通过改命令形态而不是偶然重试获得成功
- 成功差异属于可跨任务复用的高风险类别，例如：
  - 编码
  - 引号
  - 路径
  - 遍历或搜索
  - 工具发现
  - 外部 CLI 调用骨架

## Persistent Mode

- 本线程第一次满足上述条件后，默认进入“持续回写检查”模式。
- 在该模式下，每当再次出现“先失败后成功”的高风险命令，先完成模式库回写，再继续主任务。
- 用户若明确说 `别回写`、`这步别记`、`先别更新模式库` 或等价表达，则当前步跳过；若用户说的是全线程停用，则退出该模式。

## Checklist

1. 归类失败原因。
   - 只记录形态类别，不抄失败命令全文。
2. 抽取成功命令骨架。
   - 保留 `<INPUT_PATH>`、`<TOOL>` 这类占位符，不写具体路径。
3. 找承载位置。
   - 先看 [validated-command-patterns.md](validated-command-patterns.md) 的最近场景。
   - 若已有相近条目，优先更新现有文件。
   - 只有确实没有承载位时，才新增窄场景文件。
4. 回写最小信息。
   - 至少补 `validated_shape`
   - 视需要补 `preflight`
   - 在 `avoid` 或 `capture_rule` 中点明此前失败的形态类别
5. 向用户交代。
   - 简短说明“这次成功形态已写回模式库”，并指出更新了哪个文件。

## Boundary

- 不把线程内所有报错都强行沉淀。
- 不为了写库而延迟主任务太久；回写应保持最小、直接、可复用。
- 不把项目私有路径、文件名、用户名写进模式库。
