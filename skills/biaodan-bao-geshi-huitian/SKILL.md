# 表单保格式回填

## Purpose

这个 skill 用来处理“现成 Word 表单定点回填”的场景。

重点不是把文字填进去，而是同时满足这四件事：

- 内容正确
- 原件不被误改
- 字体、字号、表格、段落和页面版式尽量保持不变
- 结果有读回验证，而不是只看命令有没有跑完

它尤其适合这类任务：

- 团组织推优表、入党审核表、登记表、推荐表、审批表
- 用户已经给了一个现成 `.doc` 或 `.docx` 模板
- 用户明确说“按原格式填写”“不要动版式”“只回填内容”
- 老 `.doc` 位于同步盘、网盘、共享目录等容易导致 Word 保存异常的位置

## Core Principle

先保格式，再填内容。

默认遵守这些原则：

1. 优先回填原始 Word 模板的副本，不重建新表。
2. 默认不改原件，只生成 `-预填版`、`-终稿` 或用户指定名称的副本。
3. `.doc` 老格式如果在同步盘目录里保存不稳定，先复制到本机临时目录操作，保存成功后再拷回目标目录。
4. 正式批量回填前，先做最小保存探针：改一个小单元格、保存、读回，确认链路可用后再全量写入。
5. 写入后必须读回关键字段和关键格式，不凭“命令执行成功”就宣称完成。

## Workflow

1. 先确认任务边界。
   默认优先确认：
   - 输入文件路径
   - 是只填内容，还是也要补照片、签章占位、意见栏
   - 是否必须严格保持原 Word 版式
   - 输出是预填版还是直接终稿

2. 先抽字段，再分已知与未知。
   优先从这些来源提取信息：
   - 当前对话
   - 用户给的 Word、图片、截图
   - 同目录或指定路径下的佐证材料

   把字段分成三类：
   - 可直接回填
   - 可据材料拟写
   - 必须等待用户确认

3. 先看文档结构，不凭模板名猜。
   最少确认：
   - 段落数
   - 表格数
   - 目标字段位于段落还是表格单元格
   - 是否有合并单元格
   - 是 `.doc` 还是 `.docx`

4. 先生成副本，再操作副本。
   默认不要直接覆盖原件。
   如果用户要求改原件，先明确提醒风险再执行。

5. `.doc` 与同步盘场景优先走临时目录。
   如果满足任一条件，优先先拷到本机临时目录再操作：
   - 文件是老 `.doc`
   - 文件位于同步盘、网盘、共享盘、映射盘
   - 先前保存曾出现卡死、只读、占用、写不回

6. 正式写入前先做最小探针。
   推荐流程：
   - 复制文档副本
   - 只改一个不重要或可回退的小单元格
   - 保存
   - 重新打开并读回
   - 确认内容确实落盘、字体字号没明显跑掉

   如果探针失败，不要立刻全量回填。

7. 正式回填时优先保留原格式。
   对段落或单元格写入时，优先保留：
   - `Font.NameFarEast`
   - `Font.Name`
   - `Font.Size`
   - `Font.Bold`
   - 段落对齐
   - 段前段后间距
   - 行距规则

   如果只是替换单元格内部文字，优先在目标 `Range` 内定点写入，不重排整张表。

8. 正式写入后做最小验证。
   至少验证：
   - 关键字段是否正确读回
   - 关键意见栏是否完整读回
   - 关键单元格字体和字号是否保持模板样式
   - 输出文件是否真的生成在目标位置

9. 向用户交付时，默认同时说明：
   - 输出文件路径
   - 已填字段
   - 仍留空字段
   - 验证结果
   - 是否存在未解决风险

## Common Failures

### 1. 中文变成 `?`

这通常不是文档本身的问题，而是 shell 到 Python、PowerShell 到外部进程的编码链路丢了。

优先处理方式：

- 先统一 `UTF-8`
- 需要 Python 时优先设置 `PYTHONUTF8=1` 或使用 `python -X utf8`
- 不要在未确认编码链路正常时，直接批量生成含中文内容的中间文件

### 2. Word 打开正常，但保存卡死或写不回

常见于：

- 老 `.doc`
- 同步盘目录
- 共享目录
- 文件刚被多个 Word 进程打开过

优先改走“本机临时目录保存 -> 再拷回”的路线。

### 3. 文档被占用或只读

先区分是哪一个文件被占用：

- 原件
- 旧的预填版
- 当前临时副本

不要反复对同一路径硬写。

优先动作：

- 先确认 Word 是否还开着
- 如无必要，不强制杀所有 Word
- 若确需关闭全部 Word 进程，先向用户说明这会关闭其正在编辑的全部 Word 文件

### 4. 用 COM 写进去了，但格式跑掉

说明写入方式过粗。

优先修正为：

- 只改目标 `Range`
- 写入前先读出原有字体和段落参数
- 写入后把原格式参数补回目标范围

### 5. `.doc` 写不稳，但 `.docx` 能写

不要直接把它重建成 `.docx` 就当完成，除非：

- 用户明确接受格式迁移
- 原始 `.doc` 确实无法安全回填
- 你已经向用户说明这是保底方案，不是等价方案

## Output Structure

默认交付结构应尽量包含这几项：

- 输出文件路径
- 这次已回填的字段
- 还需要用户补充的字段
- 验证结果
- 如果用了保底方案，要明确说是保底方案

## Boundary

- 不负责给普通长文或 Markdown 新套一层 Word 模板，这类任务交给 `word-muban-geshihua`。
- 不负责默认模板、`Normal.dotm`、模板体系治理，这类任务交给 `word-template-governance`。
- 不把“重建一个看起来相似的新文档”当成默认成功标准。
- 不在未说明风险的情况下直接覆盖原件。
- 不在文件连续失败两次且没有新证据时继续盲重试。
- 不因为用户只说“帮我填 Word”就默认可以关闭其全部 Word 进程。

## Maintenance

- 如果这类回填任务继续高频出现，且保存探针、临时目录保存、格式保持和读回验证已经稳定成固定步骤，再把可复用逻辑上提到 `scripts/`。
- 如果以后经常需要在路由层直接命中“按原格式填写 Word 表格/表单”，再考虑把自然语言触发词补进 `AGENTS.md`。
- 如果后续发现它与 `word-muban-geshihua` 的边界持续重叠，优先收紧两边 `description`，不要马上合并。
