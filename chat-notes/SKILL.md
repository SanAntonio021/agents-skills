# 对话经验整理

先把当前对话里已经证明有价值的做法整理出来，再判断这些经验该写到哪里。

## 先这样做

- 先回看当前对话，直接从已有上下文里提取信息，不要一开始就让用户重复背景。
- 如果信息还不够，就用小白能听懂的话问问题，一次只问一个。
- 根据用户回答继续追问，直到已经能判断用户真正想整理什么、想达到什么结果。
- 如果现有上下文已经够了，就直接往下做，不要只是为了流程完整而多问。

## 工作顺序

1. 回看当前对话，记下这次判断真正需要的信息：
   - 反复出现的目标、输入、输出和成功标准
   - 已经被证明有复用价值的步骤、判断规则、边界和例子
   - 哪些内容只对当前项目有用，哪些内容换个项目也成立
   - 是否需要配套 `scripts/`、`references/` 或 `assets/`
2. 先查本地有没有类似 skill：
   - 先看当前会话默认可用的 skills 里有没有相近的
   - 如果需要核对本地安装文件，再看 `C:\Users\SanAn\.cc-switch\skills\`
   - 如果后面准备正式修改，再去 `D:\BaiduSyncdisk\.agents\agents-skills-src\` 找对应源码目录
   - 如果同一个 skill 同时有源码目录和同步产物，后续一律把源码目录当成修改目标，把同步产物只当成安装镜像
   - 如果这次对话里同时出现运行时目录、源码目录和历史残留目录，先分清它们各自是什么：
     - 运行时入口：当前会话实际会读到的技能目录
     - 源码入口：后面准备正式修改时应该优先改的目录
     - 历史残留 / 兼容入口 / 工作区快照：只当背景信息，不当主入口
   - 如果同一个 skill 同时出现运行时入口和源码入口，汇报时要明确区分“当前加载来源”和“推荐修改目标”，不要混写
   - 如果本地还留着历史入口、兼容包装或已经退役的 `*-local` 变体，只把它们当背景信息，不把它们当当前主入口，也不要把它们写成推荐修改目标
   - 不只看名字，也看目标、流程和边界是不是差不多
3. 再查外面有没有类似 skill：
   - 优先参考当前会话可用的 `find-skills`
   - 按 `find-skills` 的方法查官方技能站和 GitHub 上有没有相近做法
   - 如果外部查询没有做完或网络不可用，要明确说明已经查到了什么、还缺什么
4. 把对话内容、本地情况和外部情况放在一起想清楚，再给出最终结论：
   - `整理成新 skill`
   - `修改现有 skill`
   - `写在当前项目里`
   - `这次不改`
   - 如果用户已经明确在问“该新建还是改已有”，而本地又有高重合 skill 覆盖主流程，就优先在 `修改现有 skill` 和 `整理成新 skill` 之间做判断，不要轻易退回 `这次不改`
   - 如果这次对话主要是在改规则文件、维护全局规范、整理 skill 治理、处理技能路径或归档说明文件，先判断这些经验更适合落在哪一层，不要因为用户提到“经验”“规则”或“整理”，就默认整理成新 skill
5. 如果结论是 `修改现有 skill`，继续说清楚要改什么：
   - 明确写出推荐修改的源码目录路径，而不是同步产物路径
   - 建议改哪些规则
   - 建议补哪些例子
   - 建议放到哪一节
   - 为什么现有内容还不够
6. 如果这次经验最终要落到规则文件、治理文档或现有 skill，优先先给审查稿，再正式落文件。审查稿至少要先检查：
   - 有没有逻辑不清楚的地方
   - 有没有重复
   - 有没有互相打架或边界冲突
   - 标题顺序和章节分工是否合理
7. 在用户明确批准前，只交付审查稿，不直接创建新 skill，也不直接改现有 skill。
8. 如果结论是 `整理成新 skill` 或 `修改现有 skill`，把审查稿补到能直接交给 `skill-creator` 的程度：
   - skill 名称
   - `description`
   - 什么时候该触发
   - 预期输出
   - 拟议结构
   - 是否需要 `scripts/`、`references/`、`assets/`
   - 2 到 3 个真实测试提示
9. 用户批准后，再转给 `skill-creator` 做正式整理、修改和后续测试。
10. 如果这次已经正式创建或修改了 `D:\BaiduSyncdisk\.agents\agents-skills-src\` 下的 skill 源码，先按当前可用的 `skill-creator` 对照一遍：
   - 看名称、`description`、触发场景、预期输出和主体结构是不是对得上
   - 看 `evals/evals.json`、测试目录、打分结果、汇总结果和评审页是不是补齐了
11. 如果一对照发现还没对上，不要只停在“还差什么”：
   - 先补 `SKILL.md`
   - 再补 `evals/evals.json`
   - 再补测试结果、打分结果、汇总结果和评审页
12. 补完以后，再按 `skill-creator` 检查一遍，并明确写出这次结果是：
   - `已经符合`
   - `大体符合，但还差一点`
   - `还不符合`
13. 如果用户没有明确说不要提交或不要推送，就继续把收尾做完，不要只停在“我看了一下现在能不能推”：
   - 先找对要提交的仓库。如果这次改的是某个 skill 自己的源码仓，就在那个仓库里检查、提交和推送；不要只看外面的总目录
   - 如果外面还有一层大仓库，优先用真正放 skill 源码的那个仓库；不能因为外层没有 remote，就直接说推不了
   - 如果用户已经明确说同一个仓库里的这批改动都可以一起推，就把它们当成允许一起推送的范围，不要再把它们挡成无关改动
   - 再检查当前分支、remote 和工作区状态
   - 如果这次还不符合 `skill-creator` 的要求，就先不要推送，并明确说明还差什么
   - 如果条件安全，就提交并 push 当前分支；如果条件不安全，就明确说明原因
   - push 成功后，明确提醒用户去 cc-switch 手动点一次“检查更新”
   - 不要尝试自动操作 cc-switch GUI

## 结论怎么选

- `整理成新 skill`
  本地和外面都没有足够贴合的做法，而且这次对话已经形成了稳定、可复用的流程。
- `修改现有 skill`
  已有 skill 已经覆盖主流程，只是还缺局部规则、边界、例子或触发场景。
- `写在当前项目里`
  这些经验已经值得记下来，但离开当前项目、当前环境或当前工作区就不一定成立。
- `这次不改`
  这次先不改文件，也不改 skill。常见原因只有两类：现在还不值得整理，或现有内容已经够了。如果用户明确在问“新建还是修改”，而且已经找到高重合的本地 skill，只有在你说不出任何具体修改点时，才退回这个结论。

## 汇报格式

默认按下面结构汇报：

```text
这次在做：整理经验 / 判断写到哪里

结论：整理成新 skill | 修改现有 skill | 写在当前项目里 | 这次不改

理由：
- <reason 1>
- <reason 2>

本地有没有类似 skill：
- <none / existing skill + difference>

外面有没有类似 skill：
- <none / existing skill + difference>

如果结论是修改现有 skill，建议这样改：
- <change 1>
- <change 2>

如果结论是新建或修改 skill，再给审查稿：
- skill 名称：<name>
- description：<description>
- 什么时候该触发：<when to use>
- 预期输出：<expected output>
- 拟议结构：<section 1> / <section 2> / <section 3>
- scripts / references / assets：<needed or not>
- 建议测试提示：<prompt 1> / <prompt 2> / <prompt 3>

如果这次已经正式修改并完成收尾，再补：
- skill-creator 对照检查：<符合 / 大体符合但还差一点 / 还不符合 + reason>
- 补全状态：<已经补完了什么 / 还缺什么>
- 实际提交的是哪个仓库：<repo path>
- 推送状态：<pushed / skipped / blocked + reason>
- cc-switch 提醒：请手动点一次“检查更新”
```

如果结论是 `这次不改`，直接给一句明确结论，再加 1 到 2 句必要理由，不要硬把整套模板填满。

## 例子

**例子 1：整理经验，最后去修改现有 skill**

用户说：
```text
把这段对话整理一下。我们刚才反复在修 Windows 命令的路径和引号写法，这些经验看起来更像该写进现有的命令 skill。先别改文件，先给我一个判断和审查稿。
```

你应该做：
- 先从当前对话提取可复用经验
- 先查本地和外面有没有类似 skill
- 给出最终结论，并说明为什么更适合修改现有 skill
- 如果结论是修改现有 skill，再给出建议改法和审查稿

**例子 2：整理经验，最后整理成新 skill**

用户说：
```text
把我们刚才这轮对话整理一下。我感觉已经形成了一套稳定流程，但不确定值不值得单独做成一个 skill。
```

你应该做：
- 直接从当前对话提取可复用经验
- 先查本地和外面有没有类似 skill
- 给出最终结论，并说明为什么更适合整理成新 skill、写在当前项目里，还是这次不改

**例子 3：判断经验该写到哪里**

用户说：
```text
我们刚才讨论的那些坑，哪些只该写在这个项目里，哪些离开当前项目也成立？
```

你应该做：
- 先区分哪些内容只对当前项目有用，哪些内容换个项目也成立
- 再给出这些经验更适合写到哪里
- 不要因为用户提到“经验”就默认整理成新 skill

**例子 4：用户明确在问新建还是修改**

用户说：
```text
我们已经把这套方法聊得很完整了。你先查本地和公开生态有没有相近 skill，再直接告诉我这次到底该新建，还是修改已有的。
```

你应该做：
- 如果本地已经有高重合 skill 覆盖主流程，优先落到 `修改现有 skill`
- 只有在目标、输入输出和主流程都明显不同，才落到 `整理成新 skill`
- 不要因为公开生态没有完全同类，就轻易退回 `这次不改`

**例子 5：正式改完 skill 后的收尾**

用户说：
```text
这个 skill 已经按审查稿改完了。能安全推送的话就帮我把这次相关改动推上去，最后提醒我去 cc-switch 里手动点一下检查更新。
```

你应该做：
- 先检查 git 分支、remote 和工作区状态
- 只处理这次相关的 skill 改动，不把无关文件一起提交
- 如果条件安全，就提交并 push
- push 成功后提醒用户去 cc-switch 手动点“检查更新”
- 不要尝试自动点击 GUI

## 边界

- 不要把一次性的临时说法直接整理成 skill，除非它已经形成稳定、可复用的流程。
- 不要只看名字判断有没有类似 skill；重点看目标、步骤和边界。
- 不要因为用户提到“经验”就默认要整理成新 skill。
- 不要跳过本地检查，只去看外面。
- 不要在外部查询没做完时假装已经确认完毕。
- 用户没批准前，不创建新目录，不写正式 skill 正文，也不改现有 skill 主体。
- 正式修改时，优先编辑 `D:\BaiduSyncdisk\.agents\agents-skills-src\<skill-name>\` 下的源码，不直接改 `C:\Users\SanAn\.cc-switch\skills\` 里的同步产物。
- 不要把历史入口、兼容包装或已退役的 `*-local` skill 当成当前主入口，除非用户明确点名要看它。
- 不要把无关改动一起提交或推送。
- 不要在没有 remote、工作区不安全或用户明确说不要推送时强行 push。
- 不要尝试自动操作 cc-switch GUI；这里只负责推送和提醒。

## 维护

- 这份 skill 负责整理经验、判断该写到哪里、查有没有类似 skill，并给出审查稿。
- 一旦进入正式新建或正式修改阶段，就转给 `skill-creator`。
- 当判断标准、输出模板或例子变化时，优先更新这里。
