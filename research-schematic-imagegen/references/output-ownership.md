# 输出文件所有权与非破坏性整理

科研图目录常同时保存历史定稿、其他课题图和本轮候选图。目录名称是 `final` 不表示其中所有文件都由当前任务管理，也不授权代理整理既有资产。

## 所有权判定

- 任务开始前已存在的文件视为用户资产。
- 当前任务新生成、复制或编辑出的文件才属于本轮可管理集合。
- 用户明确点名要求替换、移动、重命名或归档的既有文件，才进入本轮可管理集合。
- 无法确认来源时按既有用户资产处理。

未经用户明确授权，不得对既有用户资产执行：

- 删除
- 移动或归档
- 重命名
- 覆盖
- 改写内容或元数据

## 开始任务前

写入 `prompt/`、`working/` 或 `final/` 前，先只读列出目标目录已有文件。至少记录路径、大小和修改时间；高风险整理任务再记录哈希。

发现目标文件已存在时，不自动覆盖。使用新文件名或请求用户确认。

## 最终交付

“用户要求 N 张图”约束本轮交付清单，不约束一个已存在目录中的历史文件总数。

优先顺序：

1. 在 `record.md` 中维护本轮交付文件清单。
2. 用 `verify-output.js --manifest <manifest.json>` 只验证清单中的文件。
3. 目录存在其他 PNG 时，将其报告为 `extra_files`，不移动、不删除，也不把它们视为本轮失败。
4. 用户明确要求物理隔离时，新建任务级目录，例如 `final/<task-slug>/`。
5. 用户明确要求清理既有目录时，先列出拟移动文件和目标路径，得到确认后再执行。

manifest 示例：

```json
{
  "files": [
    "fig1_final.png",
    "fig2_final.png"
  ]
}
```

验证示例：

```powershell
node <skill-dir>/scripts/verify-output.js `
  --dir <final-dir> `
  --manifest <manifest.json> `
  --expected-count 2 `
  --width 1536 `
  --height 1024 `
  --json
```

## 允许归档的范围

可直接移入 `working/` 或本轮归档区的内容仅限：

- 本轮生成的过程候选图
- 本轮生成后被用户明确否定的版本
- 本轮创建且已被新版本替代的临时遮罩、裁剪和诊断图

历史定稿、其他课题图、来源不明文件不属于“过程版本”。

## 冲突处理

如果目录总数与用户要求不一致：

1. 不做任何文件移动。
2. 验证本轮 manifest。
3. 报告额外文件名称及其“非本轮管理”状态。
4. 只有用户明确要求目录清理时，才提出可逆的移动方案。

如果已经误移动既有文件，立即停止后续整理，保留路径映射和哈希；先向用户说明，再经用户确认执行原位恢复。
