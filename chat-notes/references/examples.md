# chat-notes 例子

## 例子 1：判断经验该写到哪里

用户说：
```text
把刚才关于项目私有经验和跨项目方法的讨论整理一下，不要直接改文件。
```

应该做：
- 从当前对话提取经验
- 区分项目私有和跨项目通用
- 分流判断：涉及跨项目方法 → 有可能做 skill → 走完整流程
- 查本地和外部类似 skill
- 给出四类结论之一和完整审查稿

## 例子 2：对话结束收尾

用户说：
```text
这轮先到这里，整理一下，把没用的中间文件也清一下。
```

应该做：
- 沉淀有价值经验
- 更新项目 README/AGENTS/CLAUDE 或标准文件
- 把中间文件和疑似垃圾移到 `archive/cleanup-YYYYMMDD/`
- 更新 `.gitignore`
- 给清理清单和 Git 状态

## 例子 3：非代码项目扁平化

用户说：
```text
这个调研目录里文档太乱，收尾整理一下。
```

应该做：
- 建立或更新 `README.md` 和 `AGENTS.md`
- 把成文说明命名为 `doc-*`
- 把台账索引命名为 `record-*`
- 把最终成果命名为 `output-*`
- 旧入口和重复文件归档

## 例子 4：正式改完 skill 后推送

用户说：
```text
这个 skill 已经按审查稿改完了，能安全推送就推上去。
```

应该做：
- 按 `skill-creator` 对照检查
- 补齐缺口
- 找对真正源码仓库
- 检查 git 和 remote
- 只提交本次相关改动
- push 后提醒 cc-switch 手动检查更新

## 例子 5：分流后直接写项目记忆

用户说：
```text
帮我把这次排查 Nginx 反向代理到 3000 端口的步骤记下来，下次同一个项目能查。
```

应该做：
- 从对话提取经验（Nginx 反向代理配置步骤）
- 分流判断：这是针对当前项目的一次性配置记录，不是跨项目可复用的工作流 → 不值得做 skill
- 跳过本地和外部 skill 搜索
- 结论：写在当前项目里
- 输出简短审查稿，说明写到哪个文件、写什么内容
