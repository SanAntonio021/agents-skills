# 技术表达合同

生图前先把技术含义写清。合同用于约束图片，不是申报书正文。

## 单图合同

```markdown
# <图名>

- 用途：基金申报 / 工程申报 / 答辩 PPT / 论文概念图 / 内部讨论
- 权威依据：<本地文件、章节、日期或用户确认内容>
- 核心主线：<一句话说明图要表达的技术关系>
- 必须出现：
  - <对象、模块、阶段或信号流>
- 可以简化：
  - <不影响技术含义的细节>
- 禁止出现：
  - <旧方案、越界职责、未经确认的功能或数值>
- 准确标签：
  - <短中文或英文标签>
- 允许的定量信息：
  - <已经确认且适合放图内的数值；没有则写“无”>
- 待确认：
  - <确认前不能让模型自行决定的问题>
```

## 系列视觉合同

```markdown
- 图片数量：<N>
- 画幅：<例如 3:2 横版>
- 像素：<例如 1536 x 1024>
- 背景：纯白或接近纯白，无装饰性背景
- 主色：深蓝、蓝灰、黑灰
- 强调色：最多 1-2 种，只表示风险、关键节点或闭环结果
- 标题：同一位置、字号和层级
- 模块：统一边框、圆角、阴影和图标风格
- 连线：主信号流最突出，辅助控制流次之
- 文字：只保留短标签，不放说明段落
- 最终目录：只保留 N 张当前选定图
```

## 技术复核

逐项回答“是/否/不适用”：

- 图中每条主箭头都能在技术依据中找到对应关系。
- 图中没有模型自行增加的系统能力。
- 图中没有把备选方案画成核心实现。
- 图中没有把后续课题或实验验证职责画进当前课题。
- 所有数值、频段、速率、编码、阵列规模和距离均有依据。
- 标签和当前术语表一致。
- 图片仍符合最新技术方案；若不符合，已标为候选或待修正，而非最终图。
